# FAQ

### Why does result normalization happen in code rather than the LLM?

Technically the LLM could generate a natural language summary after receiving the tool result, but the current approach keeps normalization in code for four reasons:

1. **The frontend renders structured data, not text.** The tables (address objects, address groups, members) are rendered from the raw JSON arrays returned by the tool. If the LLM produced a text summary instead, the rich table display would be lost entirely.

2. **`direct_answer` is supplementary, not a replacement.** It is a badge shown *above* the tables. Code normalization lets both coexist — a one-liner human summary with the full structured detail rendered below it.

3. **Reliability.** `llama3.1:8b` has shown it occasionally narrates instead of calling tools. Adding a second LLM call after the tool result creates a second failure point — it could hallucinate, rephrase incorrectly, or time out.

4. **Latency.** Each LLM call takes several seconds. A second call after the tool result would roughly double the perceived response time for every query.

The LLM handles *tool selection and argument extraction* — the reasoning part. Code handles the *presentation layer* — formatting known fields from a known schema into a consistent, deterministic string.

### What is Zustand?

Zustand is a lightweight state management library for React. In standard React, components that need to share data must pass it through props from a common parent, which gets unwieldy as the app grows. Zustand solves this with a **global store** — a plain JavaScript object outside React's component tree that any component can read from or write to directly:

```js
// chatStore.js — define the store once
export const useChatStore = create((set, get) => ({
  messages: [],
  isLoading: false,

  sendMessage: async (text) => {
    set({ isLoading: true })
    // ... call API ...
    set({ messages: [...get().messages, response], isLoading: false })
  }
}))

// Any component, anywhere in the tree — no prop drilling
const { messages, isLoading, sendMessage } = useChatStore()
```

**Why Zustand over the alternatives:**

- **Redux** requires significant boilerplate (actions, reducers, dispatchers). Zustand achieves the same result in a fraction of the code.
- **React Context** re-renders every consuming component whenever anything in the context changes. Zustand components only re-render when the specific slice of state they read changes.
- **Zustand** needs no provider wrapper around the app — the store is just imported and used.

In Atlas, `chatStore.js` holds all chat state: `messages`, `conversationHistory`, `isLoading`, `currentStatus`, the `AbortController` for stopping requests, and all async actions (`sendMessage`, `loadConversations`, etc.). Components like `ChatInput`, `ChatMessages`, and `AppSidebar` all read from the same store without any prop drilling.

### Why is the session TTL 30 minutes, and can it be changed?

30 minutes was chosen as a balance between security and usability for a network tooling application:

- **Security posture** — Atlas has access to sensitive network infrastructure (Panorama, NetBox, Splunk, path analysis). A short TTL limits the window of exposure if a session cookie is stolen or a workstation is left unattended.
- **Usage pattern** — Most queries complete in under a minute. A 30-minute window comfortably covers active use without constant re-authentication, while still expiring idle sessions.
- **Alignment with enterprise standards** — 30 minutes is a common idle timeout for internal tooling in security-conscious environments.

To adjust it, change `OIDC_SESSION_TTL` in [auth.py](../../auth.py):

```python
# auth.py
OIDC_SESSION_TTL = 1800  # seconds — change to e.g. 3600 for 1 hour, 900 for 15 minutes
```

This value is used in two places: `_session_serializer.loads(..., max_age=OIDC_SESSION_TTL)` for cookie validation, and `SESSION_MAX_AGE_OIDC` in [app.py](../../app.py) for the `Set-Cookie: max-age` header. Both are driven by the same constant so changing it in one place is sufficient.

### Where is the tool schema defined?

The tool schema is not written manually anywhere — it is auto-generated by FastMCP at startup from two sources in each tool function:

**1. The function signature → `parameters` (JSON Schema)**

FastMCP uses Python type hints to build the `inputSchema` automatically:

```python
@mcp.tool()
async def query_panorama_ip_object_group(
    ip_address: str,
    device_group: Optional[str] = None,
    vsys: str = "vsys1"
) -> Dict[str, Any]:
```

FastMCP sees `ip_address: str` (required), `device_group: Optional[str] = None` (optional), `vsys: str = "vsys1"` (optional with default) and generates:

```json
{
  "type": "object",
  "properties": {
    "ip_address":   { "type": "string" },
    "device_group": { "type": "string" },
    "vsys":         { "type": "string", "default": "vsys1" }
  },
  "required": ["ip_address"]
}
```

**2. The docstring → `description`**

The full docstring becomes the tool description. `chat_service._to_openai_tool()` then trims it — everything from `Args:` onward is stripped, keeping only the "Use for / Do NOT use for / Examples" sections (capped at 600 chars) — before passing it to the LLM.

The source of truth for every tool schema is the function signature + docstring at the `@mcp.tool()` definition in its respective file:

### What exactly is sent to the LLM?

For a query like `"What address group is 11.0.0.1 part of?"`, the LLM receives three things:

**1. System message** (hardcoded in `_build_llm_messages`):
```
You are a network infrastructure assistant.
Always call a tool — never answer from memory or prior context.
Tool selection rules:
short rack IDs like 'A4', 'B2' → get_rack_details;
device names with dashes like 'leander-dc-leaf1' → get_device_rack_location;
IP addresses → query_panorama_ip_object_group or get_splunk_recent_denies;
address group names → query_panorama_address_group_members;
list/all racks → list_racks.
...
```

**2. Human message** (the user query):
```
What address group is 11.0.0.1 part of?
```

**3. Tools list** (bound via `bind_tools`, one entry per registered MCP tool):
```json
[
  {
    "type": "function",
    "function": {
      "name": "query_panorama_ip_object_group",
      "description": "Find which Panorama address groups contain a given IP address.\n\nUse for: queries with an IP address asking which address group it belongs to.\nDo NOT use for: device names, group member listing, path/traffic checks.\n\nExamples:\n- \"what address group is 10.0.0.1 in?\" → ip_address=\"10.0.0.1\"",
      "parameters": {
        "type": "object",
        "properties": {
          "ip_address":   { "type": "string" },
          "device_group": { "type": "string" },
          "vsys":         { "type": "string" }
        },
        "required": ["ip_address"]
      }
    }
  },
  // ... all 8 registered tools
]
```

**What is NOT sent:**
- Prior conversation history (`historySlice = []` — each query is stateless)
- The tool result (that only arrives after the LLM picks the tool)
- Any credentials or network data

The LLM's only job at this point is to look at the system prompt + user message + tool descriptions and output a structured tool call: `{"name": "query_panorama_ip_object_group", "args": {"ip_address": "11.0.0.1"}}`.

### How are prompts generated and used?

Three inputs are assembled into the LLM message list by `_build_llm_messages()` in [chat_service.py](../../chat_service.py):

```
[ SystemMessage, ...history (last 10)..., HumanMessage(user query) ]
```

**1. System message — hardcoded routing rules**

A static string that tells the LLM which tool to pick for which query pattern:

```
"You are a network infrastructure assistant.
Always call a tool — never answer from memory or prior context.
Tool selection rules:
short rack IDs like 'A4', 'B2' → get_rack_details;
IP addresses → query_panorama_ip_object_group or get_splunk_recent_denies;
address group names → query_panorama_address_group_members; ..."
```

This never changes at runtime. It is the primary mechanism for disambiguating between tools that could both match a query.

**2. Conversation history — sanitized, last 10 messages**

Prior messages are included but raw tool results are stripped before reaching the LLM — if they were included, the model could answer the *next* query from memory instead of calling a tool:

```python
if isinstance(content, dict):
    content = content.get("message") or content.get("reason") or "[Result shown above]"
```

Only plain text is passed. The LLM never sees previous Panorama, NetBrain, or Splunk data.

**3. Human message — the user's query verbatim**

Appended last: `HumanMessage(content=prompt)`.

**Tool descriptions — the second type of prompt**

Each MCP tool's docstring is converted into an OpenAI-format tool description by `_to_openai_tool()`. Everything from `Args:` onward is stripped and the result is capped at 600 characters — only the "Use for / Do NOT use for / Examples" portions reach the LLM. These are bound via `bind_tools(openai_tools, tool_choice="required")`, which forces the model to always output a structured tool call rather than free text.

**The prompt is used a second time after the tool runs**

`_normalize_result(tool_name, result, prompt)` uses the original user query again — not to call the LLM, but for keyword matching to decide which badge to generate:

```python
if "space utilization" in prompt_lower:
    metric_answer = f"Space utilization of {rack} is {util_value}%"
elif "available" in prompt_lower and "unit" in prompt_lower:
    metric_answer = f"{rack} has {height - occupied} available units"
```

**On tool error — retry with a ToolMessage**

If the tool returns an error, the error is appended to the message list and the LLM is re-invoked (up to 3 times). The model sees the error and can select a different tool or different arguments:

```python
messages = messages + [ai_msg, ToolMessage(content=json.dumps(result), tool_call_id=...)]
# LLM re-invoked with the error in context
```

**Summary:**

```
User query
    │
    ▼
_build_llm_messages()
    ├── SystemMessage  (hardcoded routing rules)
    ├── history[-10:]  (plain text only — no raw results)
    └── HumanMessage   (user query verbatim)
    │
    ▼
LLM + bind_tools()     (tool descriptions from docstrings, trimmed to 600 chars)
    │
    ▼  → tool name + args
call_mcp_tool()
    │
    ▼  → raw result dict
_normalize_result(tool_name, result, prompt)
    │             ↑ prompt reused here for keyword-based badge generation
    ▼
JSON response → frontend
```
